
Least frequently used 
将**使用频率最低的**页面替换下来的页面置换算法

LFU（Least Frequently Used，**最不经常使用**）是一种经典的缓存淘汰算法，核心逻辑是“当缓存空间不足时，优先移除缓存中被访问次数最少的元素”，适用于“访问频率越高的元素，未来被再次访问概率越大”的场景。


### 核心原理
1. **频率统计**：为缓存中的每个元素维护一个“访问计数器”，每次元素被访问（读/写）时，计数器数值加 1；
2. **淘汰判断**：当需要新增元素但缓存已满时，遍历所有元素，筛选出“访问计数器数值最小”的元素并删除，腾出空间给新元素；
3. **平局处理**：若多个元素计数器数值相同（即“访问频率相同”），通常会结合其他规则（如“最早进入缓存”的 LRU 逻辑），优先淘汰“频率相同且最久未使用”的元素，避免随机淘汰导致的缓存命中率下降。


### 典型应用场景
- **基础缓存设计**：适用于访问模式相对稳定、高频访问元素长期固定的场景（如静态资源缓存、热门数据查询结果缓存）；
- **数据库优化**：部分数据库的缓冲池（如 MySQL 的 Buffer Pool）会结合 LFU 思想，优先保留被频繁读取的页；
- **分布式缓存**：早期 Redis（3.0+）曾通过插件支持 LFU，后续官方版本（4.0+）将其纳入核心淘汰策略之一，用于应对“热点数据集中”的场景。


### 优缺点
| 优点                                   | 缺点                                          |
| ------------------------------------ | ------------------------------------------- |
| 符合“高频元素优先保留”的直觉，缓存命中率较高（针对频率相关的访问模式） | 需额外维护“访问计数器”，增加内存开销；频繁更新计数器也会带来一定性能损耗       |
| 逻辑相对简单，易理解和实现基础版本                    | 对“突发访问”不友好（如某低频元素突然被大量访问，因历史计数器低，可能刚缓存就被淘汰） |
